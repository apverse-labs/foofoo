Modifiedre-onboarding.repository.ts
▾
What this does
The single place all of a household's onboarding answers get saved — location, diet rules, family members, cooking habits, health needs. It's also what fetches the question lists for each onboarding screen (states, main cohort cards, sub-cohort options) and tracks how far through the 9 steps a household has gotten, so they can resume if they leave partway through.
Files / tables in this module
re-onboarding.repository.ts — 18 functions covering: fetching question lists (states, cohort cards), saving each answer type (location, diet, household members, health), and marking onboarding complete.
deriveNonvegMode() — a helper that turns "how often do you eat non-veg" answers into one of 6 standard categories (veg, vegan, jain, egg-only, occasional or regular non-veg) used everywhere else in the engine.
How it helps the app
Without this, every onboarding screen would talk to the database directly in its own way — any change to how an answer is stored would mean hunting through every screen. Centralising it here means one place to fix or extend.

Modifiedre-dish-expander.repository.ts
▾
What this does
This is the "which dish should we actually suggest" engine. For a given meal slot, it scores every eligible dish against the household's diet rules, region, city type, cooking ability, and learned taste profile, then returns the best-ranked candidates.
Files / tables in this module
isDietCompatible() — hard filter that removes any dish the household can't eat (e.g. non-veg dish for a veg household) before scoring even starts.
regionAffinityScore() / cityLifestyleScore() — boost dishes that fit the household's state and city type (e.g. quick dishes score higher for busy metro households).
cookCapabilityScore() — checks a dish's cooking complexity against how much cooking help the household has.
computeDishScore() — combines every signal above into one ranking score per dish.
fetchTodayDishCandidates() — the function screens actually call to get today's ranked suggestions.
How it helps the app
This is what stops FooFoo suggesting a 45-minute non-veg recipe to a Jain household with no time to cook. Every other part of the engine (today's plan, swap suggestions) calls into this scoring logic rather than re-implementing it.

Modifiedre-cohort-resolver.repository.ts
▾
What this does
Works out which "neighbourhood profile" (cohort) a household belongs to — combining their home state, the city they're currently living in, any health conditions, and how much cooking help they have — and produces a confidence score for that match.
Files / tables in this module
resolveCityDestinationGroup() — turns a free-text city name into a standard "destination group" (e.g. home-state Tier-1 city, cross-state metro, or generic PG/hostel) by checking it against known city lists in priority order.
buildOverlayPersonaIds() — layers in extra personas for health conditions or limited cooking help, on top of the base region match.
runCohortAssignment() — the orchestrator that runs the whole process end-to-end for one household and saves the result.
How it helps the app
This is what lets a Bangalore-based Tamil household and a Mumbai-based Tamil household get different meal suggestions, even though they answered the same diet questions — the cohort captures where they actually live, not just where they're from.

▾
What this does
Builds and stores a household's 7-day meal-class plan for the current week — which meal class (e.g. "stuffed flatbread", "rice + curry") is scheduled for each slot on each day.
Files / tables in this module

pickOverlaySlots() — decides which days get a "special" meal class injected on top of the base plan (e.g. more region-specific dishes for households with a strong regional preference).
generateUserWeeklyPlan() — builds and saves the full 7-day plan.
fetchUserWeeklyPlan() — returns the already-generated plan for this week, so it isn't rebuilt on every screen visit.
How it helps the app
This is what makes the weekly plan screen consistent across the week instead of re-rolling suggestions every time a household opens the app.

Modifiedre-feedback.repository.ts
▾
What this does
Turns a household's actions (locking a dish, accepting it, swiping it away, marking "not today" or "never again") into adjusted taste scores, so future suggestions get better the more a household uses the app.
Files / tables in this module
RE_SIGNAL_WEIGHTS — the table of how much each action moves the score (locking a dish: +0.40, accepting: +0.25, etc).
computeVarietyPenalty() — docks a dish's score if it was suggested in the last 3 days, so households don't see the same meal on repeat.
isOnCooldown() — checks whether a "not today" dish is still in its 3-day cooldown window.
recordFeedback() — saves every swipe/lock/accept action as it happens.
How it helps the app
This is the learning loop — it's the difference between an app that suggests the same generic dish list forever, and one that gets noticeably more relevant to each household over a few weeks of use.

DBPersona / segment / state planner tables
▾
What this does
Three tables added in one migration, explicitly additive (no existing tables touched): a lookup keyed by persona for which meal classes fit which day/slot, a lookup keyed by household segment for add-on rules, and a lookup keyed by state for regional meal-class preference.
Files / tables in this module

How it helps the app
These replace the old tables that were shaped like the original Excel workbook the rules came from — looking up a plan used to mean re-deriving a household's "cohort" every time. These tables let the app look a household's plan up directly by persona, segment, or state in one step.

Key decisions log
Every significant architectural decision made, and why.
Why replace the old cohort-keyed lookup tables with persona/segment/state tables?
The old tables were shaped like the original Excel workbook and required re-deriving a household's "cohort" on every read. The new tables are keyed directly by persona, so the app can look up a household's plan in one step instead of several.

Why archive the old planner tables instead of deleting them?
They were fully replaced and confirmed unused by the app, but renaming them to "_archive" instead of deleting keeps a safety net and an audit trail in case anything was missed. The founder explicitly approved this one-time exception to the "never delete data" rule.

Why move onboarding screen text out of the database and into a code config file?
The wording for the 9-step onboarding quiz was being read from the database on every screen load. Moving it to a static file means the app loads faster and the text can be reviewed/changed without touching the database.
S1 · Jun 18
Why did 5 database columns get renamed to end in "FounderInfoOnly"?
Those columns hold internal planning notes that were never meant to be shown in the app, but their old names didn't make that obvious. The new name makes it impossible to accidentally wire them into a screen.
S1 · Jun 18
Why did 8 server functions get a new security check?
Those functions trigger batch jobs (like regenerating every user's meal plan) and had no protection — anyone who found the URL could call them. They now require a secret key that only the server itself has.

Why did the Play Store build format change?
Google Play stopped accepting the old "APK" build format in 2021. The build pipeline now produces the "app bundle" format Google requires.

Why fix the dotfiles skill templates, not just this one project's KNOWLEDGE.html?

New 9-step household quiz
▾
What this does
A new household answers 9 questions — diet type, family size, region, cooking time available, and so on. Their answers decide which of the 41 "personas" they're matched to, which in turn decides what meal suggestions they get.
Flow through it
User answers quiz→ Server scores against personas→ Household assigned a persona + confidence score
Key line of code
confidence NUMERIC(4,3) ← Every match also stores a 0–1 confidence score, so low-confidence matches can be flagged for review later
Why store a confidence score at all?
If the quiz answers don't clearly match any one persona, the app can fall back to safer, more general suggestions instead of confidently guessing wrong.
Modified re-onboarding-content.ts
▾
What this does
Holds every word of text shown across the 9 onboarding screens. Previously this text lived in the database and was fetched fresh on every screen; now it ships with the app itself.
Flow through it
Screen loads→ Reads text from config file→ No database round-trip needed
Key line of code
export const RE_ONBOARDING_CONTENT ← One file holding all onboarding screen wording, instead of 5 separate database columns
Why move text out of the database?
Onboarding text doesn't change per-user — it's the same for everyone. Keeping it in code is faster to load and easier for the team to review and edit.
DB Persona / segment / state planner tables
▾
What this does
Three new tables — re_persona_slot_plan, re_segment_addon_rule, re_state_class_affinity — replace the old lookup tables that were shaped exactly like the original Excel spreadsheet the meal-planning rules came from. The new tables are organised by persona, segment, and state instead, which is how the app actually needs to look things up.
Flow through it
Household has a persona→ Look up persona's meal-class plan directly→ Weekly plan generated
Key line of code
re_persona_slot_plan — 1,148 rows, 41 personas ← Every persona now has its own ready-made meal-class plan, no extra lookups needed
Why was this change needed?
The old tables forced the app to re-derive which "cohort" a household belonged to on every single read, which was slow and fragile. The new structure looks a household's plan up directly by persona in one step.
Why archive the old tables instead of deleting them?
As a safety net — they were renamed to "_archive" and kept, in case anything downstream was still reading them. The founder explicitly signed off on this one-time exception to the project's "never delete data" rule.
Modified Server function security + error fixes
▾
What this does
A full audit of all 14 server functions found 26 gaps — the most serious being 8 batch-job functions (the ones that regenerate every user's meal plan, send notifications, etc.) that had no security check at all. Anyone who guessed the URL could trigger them. All 26 gaps were fixed and deployed.
Flow through it
Audit all 14 functions→ Fix 26 gaps (auth, error format)→ Add 30 automated tests→ All tests pass
Key line of code
{ "success": false, "error": { "code": "...", "message": "...", "retry": false } } ← Every server function now fails in this exact shape, so testing and debugging is consistent across all of them
Why did missing-login errors come back as a generic server error before?
The security check was written inside the error-handling block instead of before it, so a missing login looked like a server crash (500) instead of "please log in" (401). Moving the check earlier fixed this everywhere.

Key decisions log
Every significant architectural decision made, and why.
Why replace the old cohort-keyed lookup tables with persona/segment/state tables?
The old tables were shaped like the original Excel workbook and required re-deriving a household's "cohort" on every read. The new tables are keyed directly by persona, so the app can look up a household's plan in one step instead of several.
S1 · Jun 18
Why archive the old planner tables instead of deleting them?
They were fully replaced and confirmed unused by the app, but renaming them to "_archive" instead of deleting keeps a safety net and an audit trail in case anything was missed. The founder explicitly approved this one-time exception to the "never delete data" rule.
S1 · Jun 18
Why move onboarding screen text out of the database and into a code config file?
The wording for the 9-step onboarding quiz was being read from the database on every screen load. Moving it to a static file means the app loads faster and the text can be reviewed/changed without touching the database.
S1 · Jun 18
Why did 5 database columns get renamed to end in "FounderInfoOnly"?
Those columns hold internal planning notes that were never meant to be shown in the app, but their old names didn't make that obvious. The new name makes it impossible to accidentally wire them into a screen.
S1 · Jun 18
Why did 8 server functions get a new security check?
Those functions trigger batch jobs (like regenerating every user's meal plan) and had no protection — anyone who found the URL could call them. They now require a secret key that only the server itself has.
S1 · Jun 18
Why did the Play Store build format change?
Google Play stopped accepting the old "APK" build format in 2021. The build pipeline now produces the "app bundle" format Google requires.

What this does
A new household answers 9 questions — diet type, family size, region, cooking time available, and so on. Their answers decide which of the 41 "personas" they're matched to, which in turn decides what meal suggestions they get.
Flow through it
User answers quiz→ Server scores against personas→ Household assigned a persona + confidence score
Key line of code
confidence NUMERIC(4,3) ← Every match also stores a 0–1 confidence score, so low-confidence matches can be flagged for review later
Why store a confidence score at all?
If the quiz answers don't clearly match any one persona, the app can fall back to safer, more general suggestions instead of confidently guessing wrong.

What this does
Three new tables — re_persona_slot_plan, re_segment_addon_rule, re_state_class_affinity — replace the old lookup tables that were shaped exactly like the original Excel spreadsheet the meal-planning rules came from. The new tables are organised by persona, segment, and state instead, which is how the app actually needs to look things up.
Flow through it
Household has a persona→ Look up persona's meal-class plan directly→ Weekly plan generated
Key line of code
re_persona_slot_plan — 1,148 rows, 41 personas ← Every persona now has its own ready-made meal-class plan, no extra lookups needed
Why was this change needed?
The old tables forced the app to re-derive which "cohort" a household belonged to on every single read, which was slow and fragile. The new structure looks a household's plan up directly by persona in one step.
Why archive the old tables instead of deleting them?
As a safety net — they were renamed to "_archive" and kept, in case anything downstream was still reading them. The founder explicitly signed off on this one-time exception to the project's "never delete data" rule.

What this does
A full audit of all 14 server functions found 26 gaps — the most serious being 8 batch-job functions (the ones that regenerate every user's meal plan, send notifications, etc.) that had no security check at all. Anyone who guessed the URL could trigger them. All 26 gaps were fixed and deployed.
Flow through it
Audit all 14 functions→ Fix 26 gaps (auth, error format)→ Add 30 automated tests→ All tests pass
Key line of code
{ "success": false, "error": { "code": "...", "message": "...", "retry": false } } ← Every server function now fails in this exact shape, so testing and debugging is consistent across all of them
Why did missing-login errors come back as a generic server error before?
The security check was written inside the error-handling block instead of before it, so a missing login looked like a server crash (500) instead of "please log in" (401). Moving the check earlier fixed this everywhere.

Issue 1 — Eligible Dish Pool = 0 (P001–P043, 43 personas)

Affects all non-vegan personas. The dish query succeeds and returns rows, but after the RE scoring filters (diet type + meal slot compatibility), zero dishes are eligible for any slot.

QuestionAnswer
What happenedPlan generator ran, fetched dishes, applied scoring filters — but 0 dishes matched the breakfast / lunch / dinner slot requirements. A fallback assigned the first available dish, so the plan was saved, but the carousel was empty.
WhyThe meal_types column on the dishes table is either NULL or contains values that don't match what the RE scoring engine expects (e.g. "breakfast", "lunch", "dinner", "all"). Without this tag, no dish passes the slot filter.
What the user seesApp shows a plan with dishes, but they may not be contextually appropriate (breakfast dishes at dinner, etc.).
How to fixPopulate the meal_types column for all 818+ dishes in the dishes table. Use UPDATE dishes SET meal_types = ARRAY['lunch','dinner'] WHERE ... style SQL grouped by dish type (e.g. rice, curry, dal = lunch/dinner; idli, poha, upma = breakfast; snacks = all).
PriorityHIGH — fix first — single change unblocks 43 of 50 personas

Issue 2 — No Vegan Dishes in Database (P044–P050, 7 personas)

All 7 vegan personas hard-fail. The dish query itself returns 0 rows because no dishes have diet_type = 'vegan'.

QuestionAnswer
What happenedEdge function returned HTTP 422 "No eligible dishes match the diet/allergen filters" — plan not generated at all.
WhyThe dishes table contains no rows where diet_type = 'vegan'. Vegan is a sub-category of vegetarian — many veg dishes are technically vegan but are not tagged as such.
How to fixTwo options: (A) Add dishes with diet_type = 'vegan', or (B) run UPDATE dishes SET diet_type = 'vegan' WHERE ... for dishes that contain no dairy/egg ingredients. Option B can be automated via the derive-dish-attributes Edge Function.
PriorityHIGH — vegan is a supported user diet type in the app

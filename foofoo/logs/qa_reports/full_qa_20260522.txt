🔴 CRITICAL — Must fix before any user touches the app

1. RE Jain hard filter is incomplete — non-Jain dishes leak into carousel
   Where: supabase/functions/generate-daily-plan/index.ts:140
          supabase/functions/generate-daily-plans-batch/index.ts:96
          supabase/functions/regenerate-slot/index.ts:146
   Impact: A Jain user (food_pref='jain') currently sees 15 of 24 carousel
           items as non-Jain dishes (e.g. Paneer Butter Masala with onion +
           garlic). Today's locked breakfast for that user is "Idli Sambar"
           (is_jain=false). Direct dietary harm; trust-killing for the most
           sensitive user cohort.
   Root cause: Filter is `.in('diet_type', ['veg','jain'])` with NO
               `.eq('is_jain', true)`. Because diet_type='veg' dishes are
               kept, anything tagged is_jain=false still passes.
   DB evidence: 2 active plan-level violations on 2026-05-23 for user
                7b53646c-... — both Idli Sambar (id=4) and Paneer Butter
                Masala (id=18).

2. Dish-attribute data integrity bug — 14 dishes flagged is_jain=true
   contain onion or garlic ingredients
   Where: Data side (derive-dish-attributes script or seed). Examples:
          id=3 Upma, id=6 Vada Pav, id=8 Chole Bhature, id=11 Palak Paneer,
          id=12 Rajma Chawal, id=15 Pav Bhaji, id=20 Methi Thepla,
          id=600 Kadai Paneer, id=575 Kadhai Paneer, id=878 Paneer Do Pyaza.
   Impact: Even after fixing the RE filter, these 14 dishes would still
           leak into Jain plans because they are *labelled* Jain-safe
           while violating the rule. The auto-derivation pipeline is
           clearly not working for is_jain.
   Root cause: is_jain auto-derive logic missing or not run on these rows.
               14 / 818 active dishes = 1.7% — too high to ignore.

3. planner_carousel.re_score is NULL for every row (0 / 192 populated)
   Where: generate-daily-plan/index.ts:206-213, batch:147-154, regen:212-218
   Impact: Analytics, debugging, and post-hoc verification of RE quality
           are blind. We cannot tell why a dish ranked #1 vs #7.
   Root cause: scoreDish() computes the score, ranks, then inserts only
               [planner_id, meal_slot, ref_type, ref_id, position] —
               the score field is silently dropped.

4. recommendation_debug_log is empty (0 rows since project start)
   Where: generate-daily-plans-batch/index.ts has no insert; only the
          single-plan function writes them.
   Impact: 5AM CRON, which generated 7 of 8 current plans, leaves zero
           audit trail. We cannot reconstruct RE decisions for any
           batched user.

5. Home-state regional boost not implemented
   Where: All three RE functions — no read of region_food_affinity table
   Impact: Spec step 5 missing entirely. RE delivers identical results to
           Mumbai vs Chennai users on identical Tier-1 prefs.

6. Generic suggestion_logs coverage — only 3 / 10 action types ever fired
   Where: src/components/dish/MealCard.tsx + screens
   DB evidence: action counts → viewed:49, locked:3, not_today:2.
                Missing: swiped_to, swiped_past, tapped_detail,
                tapped_ingredients, never, added_to_plan, accepted.
   Impact: RE v2 cannot learn from gestures because the gestures are
           not being logged. Sprint 6's whole "RE v2" objective is
           starved of input data.

7. carousel delete + insert is non-atomic in regenerate-slot
   Where: supabase/functions/regenerate-slot/index.ts:205–218
   Impact: Two concurrent regenerate taps on the same slot can race:

   Fix direction: single upsert, or wrap in `pg` transaction.
